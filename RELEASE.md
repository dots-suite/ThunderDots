# Create a new release for ThunderDots

ThunderDots uses setuptools-scm for version management.

The package version is inferred from Git tags. The Git tag is the single source of truth for the release version.

Do not edit pyproject.toml or thunderdots/__version__.py manually to bump the version.

---

## Versioning policy

Release versions must use the following tag format:

```
bash vX.Y.Z 
```

Examples:

```
bash v0.1.0 
v0.2.0 
v1.0.0 
```

The tag v0.1.0 produces the Python package version 0.1.0.

---

## Before releasing

No version number should be manually changed in the following files:

- README.md: no version to update manually
- pyproject.toml: no version to update manually
- thunderdots/__version__.py: generated by setuptools-scm
- CITATION.cff: no version to update if using a stable citation file without a version field

The PyPI badge in the README is updated automatically after publication.

---

## Local checks

Run the full local validation process before creating the release tag.

```
# 1. Check repository status 

git status  

# 2. Install dependencies 

uv sync --all-extras --dev  

# 3. Run lint and format checks 

uv run ruff format --check 
uv run ruff check  

# 4. Run tests 

uv run pytest  

# 5. Build the documentation 

uv run mkdocs build --strict -f mkdocs/mkdocs.yml  

# 6. Build the package 

uv build  

# 7. Check package metadata 

uvx twine check dist/*  

# 8. Clean build artifacts if needed 

rm -rf dist 
```

---

## Commit release preparation changes

If the local checks required documentation, metadata, or code changes, commit them before tagging.

```
git add . 
git commit -m "Prepare release v0.1.0" 
```

If there are no changes to commit, skip this step.

---

## Create and push the release tag

Replace v0.1.0 with the target release version.

```
git tag v0.1.0 
git push origin master
git push origin v0.1.0
``` 

---

## Publish the GitHub Release

On GitHub:

```
Releases → Draft a new release → choose tag v0.1.0 → Publish release 
```

Publishing the GitHub Release triggers the release.yml workflow.

The workflow will:

1. build the Python package;
2. verify that the built package version matches the Git tag;
3. check the package metadata;
4. publish the package to PyPI using Trusted Publishing.

---

## After publication

Check that the package is available on PyPI:

```
pip install thunderdots 
```

or with uv:

```
uv add thunderdots 
```

---

## Documentation deployment

The documentation is deployed automatically by the CI workflow when changes are pushed to main.

The documentation URL is:

```
https://chartes.github.io/thunderdots/ 
```

---

## Important notes

Do not create a PyPI release manually unless the GitHub Actions release workflow fails and manual recovery is required.

Do not store PyPI credentials in the repository.

ThunderDots should use PyPI Trusted Publishing through GitHub Actions. The release.yml workflow is authorized from the PyPI project settings using:

- GitHub owner: chartes
- Repository: thunderdots
- Workflow: release.yml
- Environment: pypi
